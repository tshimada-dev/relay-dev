# Phase5-2 セキュリティチェック報告書: gemini_video_plugin/T-02

## 検査概要
Gemini Videoプラグインのコアロジック（`_analyze_video`）におけるセキュリティ検証を実施。
本コンポーネントは外部API（Gemini）とローカルファイルを扱うため、インジェクションおよびデータ保護の観点を重点的に検査した。

## 検査結果

| 項目 | 判定 | 攻撃シナリオの検討 → 判定根拠 | 修正案 |
|------|------|------|--------|
| インジェクション | **NG** | **攻撃 (Path Traversal)**: `video_path` に絶対パスや `..` を含む相対パスを指定することで、サーバー上の任意ファイルを Gemini API にアップロード可能。`plugin.py:145` でパス制限なく `Path` を生成しており、`171` 行目でそのまま `upload_file` に渡される。拡張子チェックはあるが、既存の機密ファイル（ログや設定ファイル）をリネームまたは拡張子偽装することで回避可能。 | 許可されたベースディレクトリ（例: `./data/videos`）を定義し、`Path.resolve()` を使用して入力パスがその配下にあることを厳格にチェックする。 |
| 認証・認可 | **NG** | **攻撃 (権限外ファイルアクセス)**: MCPサーバーを実行する OS ユーザーがアクセス可能なすべてのファイルが攻撃対象となる。ユーザーごとの認可やパスのサンドボックス化が行われていない。 | プラグイン設定に `allowed_base_dirs` 等を設け、アクセス可能な範囲をホワイトリスト化する。 |
| データ保護 | **NG** | **攻撃 (機密ログ出力 / 情報漏洩)**: `plugin.py:190` で `prompt` をそのまま `logger.info` で出力しており、プロンプトに含まれる機密情報がログに記録される。また `153, 158` 行目のエラーレスポンスにシステムのフルパスを含めており、内部構造が露出する。 | プロンプトのログ出力を要約（先頭数文字など）に留める。エラーメッセージからはフルパスを除去し、ファイル名のみを返すようにする。 |
| 設定・構成 | OK | `api_key` は `SecretStr` で保持され、`model_dump` から除外されている。`MAX_DEVICES` による DoS 対策も実装済み。 | — |
| 依存関係 | OK | `google-generativeai` (Apache 2.0) を使用。既知の重大な脆弱性は報告されていない（現時点）。 | — |
| **リソース枯渇 (DoS)** | **NG** | **攻撃 (巨大ファイルアップロード)**: アップロード前にファイルサイズのチェックがない。数GBのファイルを指定されると、ネットワーク帯域と API 処理時間を長時間占有し、サービスダウンを招く恐れがある。 | `os.path.getsize()` を使用して上限（例: 100MB）を設け、超過時は処理を拒否する。 |

## 総合判定

**Reject**

重大な脆弱性 3件：任意ファイルアクセス（Path Traversal）、機密情報のログ出力、およびリソース枯渇リスクを確認。これらは MCP ツールとして公開された際に、LLM 経由で悪意のある指示を与えることで容易に悪用される可能性がある。特にパスのバリデーション欠如は、サーバー上の機密情報漏洩に直結するため、Phase 5 での修正が不可欠である。

### Reject詳細

- 原因分類: 実装不備
- 差し戻し先: Phase5
- 根拠: `video_path` に対する検証が不十分であり、システムの任意ファイルにアクセス可能な状態であるため。
- 修正指示:
  1. `video_path` のアクセス範囲を指定されたディレクトリ内に限定すること（`Path.resolve()` を使用し、ベースディレクトリと比較する）。
  2. プロンプトのログ出力を制限し、機密情報の漏洩を防止すること。
  3. `upload_file` 実行前にファイルサイズを確認し、適切な上限値を設けること。
  4. エラーメッセージからシステムのフルパス情報を除去し、ファイル名のみを返却すること。

## 要約（200字以内）
T-02セキュリティチェック：Reject。NG 3件。`video_path` のバリデーション不備により、サーバー上の任意ファイルを Gemini API へアップロード・解析可能な Path Traversal 脆弱性が存在。また、ログへのプロンプト全文出力による機密情報漏洩、ファイルサイズ未チェックによる DoS リスクも確認。これらは重大なリスクであるため、Phase 5 に差し戻し、パスの制限と安全なエラーハンドリングの実装を求める。
