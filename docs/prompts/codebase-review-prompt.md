目的は、現実的なリスク評価と改善余地の特定です。
重大な欠陥がない場合は、無理にP0を作らないでください。

# レビュー方針
・欠陥を誇張しない
・実際に発生確率が低いものは明確に「低確率」と記載
・設計が妥当である部分は妥当と明言する
・改善余地と致命的欠陥を明確に区別する
・「理論上の最悪ケース」と「実運用上の現実的リスク」を分離する
・具体的な根拠はファイルパスと行番号で示すこと

# 出力フォーマット

## 1. 総評（3〜5行）
・現状の完成度
・設計の成熟度
・運用可能性

## 2. リスク評価（確率×影響度で評価）
各項目を以下で評価：
- 発生確率：高 / 中 / 低
- 影響度：重大 / 中 / 軽微
- 総合リスク：高 / 中 / 低

評価対象：
- 自律制御
- セキュリティ
- 状態管理
- テスト戦略
- デプロイ安全性

## 3. 改善提案（優先度は実害基準）
- P0：現実的に事故を起こし得るもののみ
- P1：運用品質向上
- P2：設計洗練・将来拡張

※P0が存在しない場合は「P0なし」と明記すること

## 4. 強み
・設計上評価できる点
・実務的に良い判断

## 5. 技術成熟度評価
1（試作）〜100（商用安定運用可能）
目安：50＝検証用途なら問題ない水準、80＝小規模本番運用可能、95以上＝商用安定

# 重要
・抽象的な一般論は禁止
・P0は乱発しないこと
・欠陥がない場合は明確に「重大欠陥なし」と記載すること
