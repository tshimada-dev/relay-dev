# Phase3 出力例（Few-shot）

以下は「商品検索レスポンス改善」の設計書の抜粋例。実際の出力では全8セクションを含めること。

### 1. 機能一覧（ユーザーストーリー形式）

| ID | ユーザーストーリー | 優先度 |
|----|-------------------|--------|
| F-01 | ユーザーとして、商品検索結果が500ms以内に表示されることで、ストレスなく商品を探せる | 必須 |
| F-02 | ユーザーとして、商品名と説明文の両方で検索できることで、目的の商品を見つけやすい | 必須 |
| F-03 | 運用者として、検索インデックスの状態を監視できることで、性能劣化を早期検知できる | 推奨 |

### 3. API設計

#### GET /api/products/search

- 入力:
  - `q` (string, 必須): 検索キーワード
  - `page` (int, 任意, default=1): ページ番号
  - `per_page` (int, 任意, default=20, max=100): 1ページあたりの件数
- 出力:
  ```json
  {
    "products": [{ "id": 1, "name": "...", "description": "...", "score": 0.95 }],
    "total": 150,
    "page": 1,
    "per_page": 20
  }
  ```
- 性能要件: p95 500ms以下
- キャッシュ: Redis、TTL 5分、キーは `search:{hash(q)}:{page}:{per_page}`

### 4. データ構造

#### 追加インデックス

```sql
CREATE INDEX idx_products_search ON products
  USING GIN (to_tsvector('japanese', name || ' ' || description));
```

### 7. 機能間依存関係

| 機能ID | 機能名 | 依存する機能/タスク | 依存理由 |
|--------|--------|---------------------|----------|
| F-01 | 検索高速化 | IDX（インデックス追加）, CACHE（キャッシュ層導入） | インデックスとキャッシュの両方が必要 |
| F-02 | 複合フィールド検索 | IDX（インデックス追加） | name + description のインデックスが前提 |
| F-03 | 監視 | CACHE（キャッシュ層導入） | キャッシュヒット率の監視が必要 |
| IDX | インデックス追加 | MIG（DBマイグレーション） | マイグレーション基盤が前提 |

### 8. 主要処理フロー

#### 検索APIリクエスト処理フロー

| ステップ | 処理内容 | 担当コンポーネント | 成功時 | 失敗時 |
|---------|---------|-------------------|--------|--------|
| 1 | リクエスト受信 | API Gateway | → Step 2 | 400エラー返却 |
| 2 | パラメータバリデーション | search.py | → Step 3 | 400エラー返却 |
| 3 | キャッシュ確認 | cache.ts | キャッシュヒット → Step 7 | キャッシュミス → Step 4 |
| 4 | DB検索実行 | search.py + PostgreSQL | → Step 5 | 500エラー返却 |
| 5 | 検索結果整形 | search.py | → Step 6 | — |
| 6 | キャッシュ保存 | cache.ts (TTL=5分) | → Step 7 | ログ出力のみ（処理続行） |
| 7 | JSON応答返却 | API Gateway | 完了 | — |

**エラーハンドリング**:
- Step 3でRedis障害時 → キャッシュをスキップしてStep 4へ
- Step 4でDB障害時 → 500エラー + アラート送信
- Step 6でRedis障害時 → キャッシュ保存失敗をログ記録、レスポンスは正常返却

### 9. エラーハンドリング設計

| エラー種別 | HTTPステータス | エラーコード | メッセージ例 | ログレベル | アラート |
|-----------|---------------|-------------|-------------|-----------|--------|
| バリデーションエラー | 400 | E_VALIDATION | "パラメータqは必須です" | INFO | なし |
| 認証エラー | 401 | E_UNAUTHORIZED | "認証に失敗しました" | WARNING | なし |
| 権限エラー | 403 | E_FORBIDDEN | "この操作は許可されていません" | WARNING | あり |
| リソース未存在 | 404 | E_NOT_FOUND | "指定されたリソースが見つかりません" | INFO | なし |
| レート制限超過 | 429 | E_RATE_LIMIT | "リクエスト制限を超過しました" | WARNING | あり（閾値超過時） |
| DB接続エラー | 503 | E_SERVICE_UNAVAILABLE | "サービスが一時的に利用できません" | ERROR | あり |
| 内部エラー | 500 | E_INTERNAL | "内部エラーが発生しました" | ERROR | あり |

**エラーレスポンス形式**:
```json
{
  "error": {
    "code": "E_VALIDATION",
    "message": "パラメータqは必須です",
    "details": [{ "field": "q", "issue": "required" }]
  }
}
```

**原則**:
- エラーメッセージに技術的詳細（SQL、スタックトレース）を含めない
- 内部エラーの詳細はログに記録し、ユーザーには汎用メッセージのみ返す
- リトライ可能なエラー（503）とリトライ不可（400, 403）を明確に区別

### 10. データ移行計画

**対象**: 既存商品テーブル（products、既存レコード約5万件）にGINインデックスを追加

| 項目 | 内容 |
|------|------|
| 移行対象データ | 既存の商品テーブル（products）に全文検索インデックスを追加 |
| 移行方式 | オンライン（CONCURRENTLY オプション使用） |
| ダウンタイム | なし（読み取り/書き込み継続可能） |
| ロールバック手順 | `DROP INDEX CONCURRENTLY idx_products_search;` |
| データ整合性検証 | 移行前後の全件数比較（5万件一致確認）、サンプル検索クエリで結果一致確認 |
| 移行スクリプト | `migrations/0004_add_products_search_index.sql` |
| 所要時間見込み | 約10分（5万件、PostgreSQL 14） |

**リスク対策**:
- 本番適用前にステージング環境で実施（2026-02-18予定）
- ピークタイム外（深夜2:00）に実行
- 移行前に全データのRDS自動スナップショット取得を確認

### 11. ロールバック戦略

| シナリオ | ロールバック方法 | 所要時間 | 注意事項 |
|---------|-----------------|---------|----------|
| API レスポンス遅延 | 機能フラグでキャッシュ無効化、インデックス使用停止 | 1分 | 旧検索ロジックへの自動フォールバック |
| インデックス構築失敗 | `DROP INDEX CONCURRENTLY idx_products_search;` | 5分 | 既存検索機能は影響なし |
| Redis接続エラー | キャッシュ層をバイパス（設定: `CACHE_ENABLED=false`） | 2分 | パフォーマンス劣化あり |
| 全体的な性能劣化 | Git revert + 再デプロイ + インデックス削除 | 15分 | DB変更も含むためダウンタイム5分発生 |

**ロールバック判断基準**:
- p95レスポンスタイム > 1000ms が5分継続 → 即座にロールバック
- エラー率 > 1% が3分継続 → 即座にロールバック
- ユーザー影響が軽微な場合 → 30分以内にホットフィックス適用、不可能ならロールバック

### 12. 監視・アラート設計

| 監視項目 | 計測方法 | 正常範囲 | アラート閾値 | 通知先 |
|---------|---------|---------|-------------|--------|
| 検索API p95レスポンスタイム | Datadog APM | < 500ms | > 1000ms (5分間) | Slack #alerts |
| 検索API エラー率 | Datadog Logs | < 0.1% | > 1% (3分間) | PagerDuty (Critical) |
| Redisキャッシュヒット率 | Redis INFO stats | > 80% | < 50% (10分間) | Slack #alerts |
| PostgreSQL CPU使用率 | CloudWatch RDS | < 70% | > 85% (5分間) | PagerDuty (Warning) |
| インデックスサイズ | PostgreSQL pg_indexes | — | 予想の2倍超過 | Slack #alerts |
| 検索クエリ実行時間（DB） | PostgreSQL slow query log | < 200ms | > 500ms (10回/分) | Slack #alerts |

**ダッシュボード構成**:
- リアルタイム監視: 検索API レスポンスタイム、エラー率、キャッシュヒット率、リクエスト数/秒
- 日次レポート: 検索キーワード Top 10、検索結果0件のクエリ Top 10、パフォーマンス推移

**ログ出力設計**:
- **INFO**: 検索クエリ実行（`search_query={q}, results={count}, response_time={ms}`）
- **WARNING**: キャッシュミス連続、レート制限接近（80%到達時）
- **ERROR**: DB接続失敗、タイムアウト、予期しない例外
- **ログに含めない情報**: パスワード、APIトークン、個人情報（メールアドレス、氏名等）

**アラート対応フロー**:
1. Critical アラート（PagerDuty） → 即座に担当者を招集、15分以内に初動対応
2. Warning アラート（Slack） → 1時間以内に確認、必要に応じて調査
3. Info アラート（Slack） → 翌営業日に確認
