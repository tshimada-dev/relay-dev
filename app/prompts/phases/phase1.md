# Phase1 要件定義テンプレート

このフェーズでは、Input Artifacts の内容を整理して、実装可能な要件定義を作る。

## 実行原則

- Input Artifacts を正本として読むこと
- Required Outputs に書かれた `phase1_requirements.md` と `phase1_requirements.json` のみを作成すること
- 次フェーズの判断、タスク選択、オーケストレーション状態の更新は engine が行う。自分で遷移を決めないこと
- 情報が不足している場合は勝手に補完せず、`assumptions` または `unresolved_questions` に明示すること

## Markdown 出力に含める内容

- 要件の要約
- goals
- non_goals
- user_stories
- acceptance_criteria
- visual_acceptance_criteria
- assumptions
- unresolved_questions

## JSON 出力

`phase1_requirements.json` には以下の配列を必ず含めること。

- `goals`
- `non_goals`
- `user_stories`
- `acceptance_criteria`
- `visual_acceptance_criteria`
- `assumptions`
- `unresolved_questions`

各要素は短くても意味が通る文にすること。`acceptance_criteria` は後続の設計とテストで検証できる粒度にすること。
`visual_acceptance_criteria` は、Phase0 の `design_inputs` / `visual_constraints` や external `DESIGN.md` が存在する場合に、色・余白・タイポグラフィ・コンポーネント状態・レスポンシブ挙動などを観測可能な条件へ落とし込むこと。UI変更がない場合は空配列でよい。

## 品質基準

- goals と non_goals が混ざっていない
- acceptance_criteria が観測可能で曖昧でない
- visual design input がある場合、`visual_acceptance_criteria` が空で放置されていない
- unresolved_questions が空の場合は、未確定事項が本当に残っていない

## 思考プロセス（最初に出力）

回答を作成する前に、以下の手順で思考を行ってください：
1. **事実と推測の分離**: ユーザー入力から「事実」と「不明点（推測）」を明確に区別してリストアップする。
2. **目的・KPIのドラフト**: SMART原則（具体的、計測可能、達成可能、関連性、期限）に基づき、目的とKPIの案を作成する。
3. **スコープ定義**: 「何を作るか（What）」に集中し、「どう作るか（How）」は除外するよう意識合わせを行う。

## 重要ルール

- 推測と事実を明確に区別すること（推測には「【推測】」を付与）
- 曖昧語（いい感じ、最適、早く等）は必ず定義に分解
- 実行案はまだ出さない
- **解決策の策定はPhase3で行います。本フェーズでは「何を作るか（What）」と「なぜ作るか（Why）」の定義のみに集中し、具体的な「どう作るか（How/実装詳細）」は記述対象外としてください。**
- `DESIGN.md` 由来の指針がある場合も、実装技法ではなく「画面としてどう見えるべきか」「どう振る舞うべきか」という要求に変換すること

## 出力例（Few-shot）

以下はユーザー入力「商品検索をもっと早くしたい」に対する出力例。

```
1. 事実と推測: ユーザーは「商品検索」の速度向上を求めている（事実）。現状のレスポンスタイムや離脱率は不明（要確認）。
2. 目的・KPI: レスポンスタイム（500ms以下）、離脱率の改善（現行比20%減）を目標値とする。
3. スコープ: 大規模改修（Elasticsearch導入）ではなく、既存のPostgreSQLのチューニングを優先スコープとする（仮定）。

### 目的

商品検索のレスポンスタイムを改善し、ユーザーの離脱率を低減する。

### 成功条件（KPI/評価基準）

| 条件 | 計測方法 | 目標値 |
|------|----------|--------|
| 検索レスポンス改善 | p95レスポンスタイム計測 | 500ms以下 |
| 離脱率低減 | GA4 検索結果ページ離脱率 | 現状比20%減 |

### 制約条件

- コスト: 追加インフラ費用は月額5万円以内【仮定】
- 技術: 既存のPostgreSQL全文検索を前提とする
- 時間: 明確な期限なし（要確認）

### 前提条件

- 現在の検索レスポンスはp95で2秒以上かかっている【推測】
- 検索対象は商品名・説明文・カテゴリ【推測】
- 検索インデックスの再構築は許容される【仮定】

### 入力情報の欠損リスト

- 現在のレスポンスタイム実測値
- 検索対象フィールドの確定
- 許容ダウンタイム（インデックス再構築時）

### 実行スコープ

- 含める: 検索クエリ最適化、インデックス追加、キャッシュ導入検討
- 除外: UI変更、検索アルゴリズム変更（Elasticsearch移行等）

### リスク要因

| リスク | 影響度 | 発生確率 | 対策案 |
|--------|--------|----------|--------|
| インデックス追加によるwrite性能低下 | 中 | 中（更新頻度が高い時間帯があるため） | 書き込み頻度を事前計測し閾値設定 |
| キャッシュ導入による検索結果の鮮度低下 | 低 | 高（商品情報の更新はリアルタイム性が求められるため） | TTLを短め（5分）に設定 |

### 次フェーズ用質問リスト

- 検索対象フィールドはどれか？（商品名のみ / 商品名+説明文 / 全フィールド）
- 許容ダウンタイムは？（なし / 30分以内 / 数時間可）
- Elasticsearch等の外部検索エンジン導入は検討範囲に含めるか？（Yes / No）

## 要約（200字以内）

商品検索のp95レスポンスタイムを500ms以下に改善し離脱率20%減を目指す。PostgreSQL全文検索を前提にクエリ最適化・インデックス追加・キャッシュ導入を検討範囲とし、外部検索エンジン移行とUI変更は除外。最大リスクはインデックス追加によるwrite性能低下。検索対象フィールドと許容ダウンタイムが未確定。
```

### 悪い出力例（このように書かないこと）

```
❌「検索を最適化します」→ 何をもって最適とするか未定義
❌「パフォーマンスを改善」→ 目標値がない
❌「Elasticsearchを導入すべき」→ 実行案を出している（Phase1では禁止）
❌「おそらく2秒くらいかかっている」→ 【推測】タグがない
```
